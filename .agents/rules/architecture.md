# Архитектура

Слоистый дизайн с жёстким разделением «тест ↔ работа с API». Сюда вынесено «почему так» — в коде этих объяснений нет (см. `conventions.md`).

## Слои и поток данных

```
Test  ->  <Domain>Api  ->  BaseApi (+ Specs)  ->  HTTP (REST Assured)
```

- **Test** (`tests/`) — знает только доменный клиент и `ResponseSpecification`. Ни URL, ни ключа, ни HTTP-деталей.
- **<Domain>Api** (`api/`) — знает свои роуты (`Routes`), отдаёт сырой `Response`. **Ассертов внутри нет.**
- **BaseApi** (`api/`) — выполняет запрос на общей `RequestSpecification` из `Specs`.
- **Specs** (`specs/`) — общая часть запроса/ответа в одном месте (baseUri, ключ, JSON, ожидаемый статус).
- **Config** (`config/`) — единственный источник baseURL и ключа.

Каждый слой не знает о деталях соседа снизу больше, чем нужно. Проверки — только в тестах, HTTP — только в клиентах, секреты/URL — только в конфиге.

## Принятые решения (и почему)

1. **Весь код в `src/test/java`.** Репо не собирает продакшн-jar — это тест-проект. Значит зависимости остаются `scope=test`, ничего лишнего не «отгружается». Альтернатива (framework в `src/main/java`) сочтена оверинжинирингом для тестового репо.
2. **Ключ: env-переменная → `.env` (fallback).** Локально удобно `.env`; в CI тот же код возьмёт секрет из env-переменной без правок. Ключ в git не попадает никогда.
3. **Несекретный конфиг — в `config.properties` (classpath), секрет — в `.env`.** Секрет и несекрет разнесены физически.
4. **Request/Response Specifications вместо копипасты.** `RequestSpecBuilder`/`ResponseSpecBuilder` собирают общее один раз; тесты цепляют `.spec(...)`. Убирает дублирование настроек.
5. **Service objects (API-клиенты) без ассертов.** Клиент только ходит в сеть и отдаёт `Response`. Это и есть разделение «работа с API ↔ проверки».
6. **Логи запрос/ответ только при падении** (`enableLoggingOfRequestAndResponseIfValidationFails` в `BaseTest`). Зелёные тесты — чистая консоль; красный — полная диагностика.
7. **Стек:** Java 25 (`compiler.release=25`), JUnit 5.14.4, REST Assured 6.0.0 (baseline Java 17 — безопасна на Java 25), Maven wrapper 3.9.2. Зависимости добавляются по фазам (`PROGRESS.md`), не всё сразу.

## Что откладывается (по роадмапу)

- **Модели (POJO/record) + Jackson** — Фаза 3, когда начнём проверять тело ответа. Пока проверяем только статус — модель = мёртвый код (YAGNI).
- **Генерируемые каталоги кода** (автосканеры структуры) — когда классов станет много. Сейчас маленький ручной набор `rules/` дешевле.
