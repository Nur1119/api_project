# Глоссарий — термины Nessie и стека

Описательный: нормализует термины к тому, как они называются в API/коде. Ничего не выдумывает. Поверхность API — в `api-map.md`.

## Домен Nessie (сущности API)

| Термин | Что это |
|---|---|
| **customer** | Клиент банка. Владелец аккаунтов. Есть адрес, имя. |
| **account** | Счёт клиента (checking / savings / credit card). Есть баланс, номер, nickname, rewards. |
| **transfer** | Перевод средств между двумя аккаунтами. |
| **deposit** | Зачисление на аккаунт. |
| **withdrawal** | Списание с аккаунта. |
| **bill** | Счёт на оплату (recurring/one-off), привязан к аккаунту. |
| **purchase** | Покупка у merchant, списывается с аккаунта. |
| **merchant** | Продавец (точка), у которой совершают purchase. Есть геолокация. |

Связи: `customer` → владеет → `account` → участвует в `transfer`/`deposit`/`withdrawal`/`bill`/`purchase`; `purchase` ссылается на `merchant`.

## Термины стека

| Термин | Значение в проекте |
|---|---|
| **spec (Specification)** | REST Assured `RequestSpecification` / `ResponseSpecification` — переиспользуемая часть запроса/ответа. НЕ путать с `.spec.ts` из монорепо (там Playwright). |
| **service object / API-клиент** | Класс в `api/`, инкапсулирующий вызовы одного домена (`AccountsApi`). |
| **DTO / модель** | `record` под тело запроса/ответа (Jackson). Появятся с Фазы 3. |
| **smoke-тест** | Минимальная проверка «живо ли»: статус-код без разбора тела. |
